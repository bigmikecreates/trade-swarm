# v1.0.0 — Live Trading with Training Wheels

## Goal
Deploy real capital — but conservatively. Start with 1–2% of intended capital allocation. Run shadow mode (paper and live simultaneously). Keep all safety systems active. Only scale capital after 30 consecutive live trading days with no critical incidents and performance matching paper trading.

## Gate to enter this version
- 3 months continuous paper trading, Sharpe > 1.0, drawdown < 15% ✓
- Walk-forward optimisation passed ✓
- Stress tests passed ✓
- Written human sign-off document completed ✓
- Kill switch tested from Telegram within last 7 days ✓
- Full system running stably on VPS for 30+ days ✓

## Gate to scale capital (v1.1.0+)
- 30 consecutive live trading days, no critical gate fires
- Live performance within 20% of paper performance (slippage accounted for)
- Max live drawdown < 10%
- No manual interventions required
- Second human review completed

---

## The training wheels mindset

v1.0.0 is not "going live." It is a **controlled experiment** to discover the gap between paper trading and reality. That gap always exists. Your job in v1.0.0 is to measure it, not to profit from it.

Expected surprises:
- Slippage higher than paper (especially on crypto and small-cap equities)
- Fills rejected or delayed by broker compliance checks
- Margin requirements different from what you assumed
- Tax lot tracking complications
- Funding rate charges on crypto perpetuals
- Data feed discrepancies between paper and live

None of these will kill you at 1–2% allocation. They would at 100%.

## What to build

### 1. Shadow mode — parallel paper + live

The most important feature of v1.0.0. The system runs two execution adapters simultaneously: one paper, one live. Every decision the orchestrator makes is sent to both. You can compare fills, slippage, and P&L side by side.

```python
# brokers/shadow_adapter.py
from brokers.alpaca_adapter import AlpacaAdapter
from logging.db import log_trade
from dataclasses import dataclass

@dataclass
class ShadowResult:
    live_report:  dict
    paper_report: dict
    slippage_gap: float   # live slippage - paper slippage in bps

class ShadowBrokerAdapter:
    """
    Sends every order to both live and paper.
    Records discrepancies for analysis.
    """
    def __init__(self, live: AlpacaAdapter, paper: AlpacaAdapter):
        self.live  = live
        self.paper = paper

    def execute(self, order: dict) -> ShadowResult:
        # Execute on paper first (no risk), then live
        paper_result = self.paper.place_market_order(
            order["asset"], order["size"], order["direction"])
        live_result  = self.live.place_market_order(
            order["asset"], order["size"], order["direction"])

        paper_fill = self.paper.get_order_status(paper_result.order_id)
        live_fill  = self.live.get_order_status(live_result.order_id)

        paper_price = paper_fill.get("avg_price", 0)
        live_price  = live_fill.get("avg_price", 0)
        gap_bps     = ((live_price - paper_price) / paper_price * 10000
                       if paper_price > 0 else 0)

        log_trade({**order, "mode": "live",  "filled_price": live_price})
        log_trade({**order, "mode": "paper", "filled_price": paper_price})

        return ShadowResult(
            live_report  = live_fill,
            paper_report = paper_fill,
            slippage_gap = round(gap_bps, 2),
        )
```

### 2. Micro-allocation sizing

Hard-code a capital fraction multiplier that limits live allocation to 1–2% of your intended total capital. This is separate from and on top of the existing position sizing rules.

```python
# config.py additions
LIVE_CAPITAL_FRACTION = 0.02   # only deploy 2% of intended total capital to live
# Example: if you intend to eventually trade $50,000,
# your v1.0.0 live account starts with $1,000

# In position sizer:
def calculate_live_position_size(
    account_equity: float,
    entry_price:    float,
    stop_price:     float,
    capital_fraction: float = LIVE_CAPITAL_FRACTION,
    risk_pct:       float   = 0.01,
) -> float:
    effective_equity = account_equity * capital_fraction
    return calculate_position_size(effective_equity, entry_price, stop_price, risk_pct)
```

### 4. Live performance tracker

Track and compare live vs paper performance continuously. Alert if they diverge by more than 20%:

```python
# monitoring/live_tracker.py
import redis, json
from datetime import datetime, date

r = redis.Redis(decode_responses=True)

class LivePerformanceTracker:
    def __init__(self):
        self.live_pnl_history  = []
        self.paper_pnl_history = []

    def record(self, live_pnl: float, paper_pnl: float):
        ts = datetime.utcnow().isoformat()
        r.lpush("live_pnl_log",  json.dumps({"ts": ts, "pnl": live_pnl}))
        r.lpush("paper_pnl_log", json.dumps({"ts": ts, "pnl": paper_pnl}))
        r.ltrim("live_pnl_log",  0, 9999)
        r.ltrim("paper_pnl_log", 0, 9999)

    def check_divergence(self, alert_fn) -> float | None:
        live_raw  = r.lrange("live_pnl_log",  0, 29)   # last 30 trades
        paper_raw = r.lrange("paper_pnl_log", 0, 29)

        if len(live_raw) < 10 or len(paper_raw) < 10:
            return None

        live_total  = sum(json.loads(x)["pnl"] for x in live_raw)
        paper_total = sum(json.loads(x)["pnl"] for x in paper_raw)

        if paper_total == 0:
            return None

        divergence = abs(live_total - paper_total) / abs(paper_total)

        if divergence > 0.20:
            alert_fn(
                f"⚠️ Live vs paper divergence: {divergence:.1%}\n"
                f"Live P&L: ${live_total:.2f}\n"
                f"Paper P&L: ${paper_total:.2f}\n"
                f"Investigating required before adding capital."
            )
        return divergence
```

### 5. Tighter live-only risk limits

The live gate should have stricter limits than paper. The system already has a gate — add a live-specific override layer:

```python
# risk/live_gate_overrides.py
"""
Additional restrictions applied ONLY to live orders.
These sit on top of the existing gate.
"""

LIVE_OVERRIDES = {
    "max_daily_loss_pct":    0.015,   # 1.5% vs 3% in paper
    "max_drawdown_pct":      0.08,    # 8% vs 15% in paper
    "max_position_pct":      0.05,    # 5% vs 10% in paper
    "max_notional_usd":      500,     # hard $ cap at $500 per order
    "max_orders_per_minute": 3,       # stricter runaway protection
    "news_buffer_minutes":   10,      # wider news buffer
    "vix_halt_threshold":    30,      # lower VIX threshold
}
```

### 6. Live incident log

Every anomaly in live trading must be logged and reviewed:

```sql
-- Add to sql/init.sql
CREATE TABLE IF NOT EXISTS live_incidents (
    id           SERIAL PRIMARY KEY,
    timestamp    TIMESTAMPTZ DEFAULT NOW(),
    severity     TEXT NOT NULL,    -- "info", "warning", "critical"
    category     TEXT NOT NULL,    -- "slippage", "fill_failure", "gate_fire", "divergence"
    asset        TEXT,
    description  TEXT,
    resolved     BOOLEAN DEFAULT FALSE,
    resolution   TEXT
);
```

```python
# logging/incidents.py
from logging.db import get_conn

def log_incident(severity: str, category: str, description: str, asset: str = None):
    with get_conn() as conn:
        with conn.cursor() as cur:
            cur.execute("""
                INSERT INTO live_incidents (severity, category, asset, description)
                VALUES (%s, %s, %s, %s)
            """, (severity, category, asset, description))
```

Review the incident log every single day during v1.0.0. Any unresolved critical incident blocks capital scaling.

## The 30-day live checklist

Run through this every day for 30 days before declaring v1.0.0 stable:

**Daily (5 minutes)**
- [ ] Check live P&L vs paper P&L — divergence < 20%?
- [ ] Check incident log — any new entries?
- [ ] Check drawdown — still below 8%?
- [ ] Verify kill switch still reachable via Telegram
- [ ] Check Grafana — all containers healthy?

**Weekly (30 minutes)**
- [ ] Review all trades that fired the gate — were they correct?
- [ ] Compare live slippage vs paper slippage by asset
- [ ] Check agent weight drift — any agent weight below 0.3?
- [ ] Review Postgres trade log — any anomalous fills?

**End of month**
- [ ] Calculate Sharpe ratio over live period
- [ ] Calculate live max drawdown
- [ ] Compare live performance to paper performance
- [ ] Document any strategy degradation
- [ ] Human sign-off decision: scale capital or hold at current level?
